# บทพากย์ภาษาไทยฉบับละเอียดสำหรับสไลด์ทั้ง 10 หน้า

---

## สไลด์ที่ 1 — เปิดตัว

สวัสดีครับ ยินดีต้อนรับสู่คู่มือการใช้งาน OpenCode ร่วมกับ Oh My OpenAgent ในวันนี้ เราจะมาเรียนรู้วิธีเลือกใช้งาน AI Agent ให้ถูกต้องและมีประสิทธิภาพสูงสุด ซึ่งเป็นทักษะสำคัญที่จะช่วยให้คุณเขียนโค้ดได้เร็วขึ้น ประหยัดเวลา และได้ผลลัพธ์ที่ตรงใจ หลักสูตรนี้เหมาะสำหรับนักพัฒนาที่เริ่มต้นใช้ OpenCode และอยากเข้าใจว่า Agent แต่ละตัวมีหน้าที่อะไร ควรใช้ตัวไหนเมื่อไหร่ และมีข้อควรระวังอะไรบ้าง มาเริ่มกันเลยครับ

---

## สไลด์ที่ 2 — ทำไมต้องรู้จัก Agent

ก่อนที่เราจะลงรายละเอียด มาทำความเข้าใจก่อนว่าทำไมเราต้องรู้จักเลือกใช้ Agent ให้ถูกต้อง OpenCode ไม่ใช่เครื่องมือที่มี AI ตัวเดียว แต่มี Agent หลายตัวที่ถูกออกแบบมาให้ทำงานในสถานการณ์ที่แตกต่างกัน ลองนึกภาพว่าคุณมีทีมงานหลายคน บางคนเก่งงานรวดเร็ว บางคนเก่งวางแผน บางคนเก่งคิดวิเคราะห์ลึก ถ้าคุณส่งงานที่ต้องใช้ความคิดเชิงลึกให้คนที่ถนัดงานรวดเร็ว ผลลัพธ์อาจออกมาไม่ดี และต้องแก้ไขหลายรอบ ในทางกลับกัน ถ้าคุณเลือก Agent ที่เหมาะสมกับงาน คุณจะได้งานที่เสร็จเร็ว ไม่ต้องเสียเวลาตอบคำถามซ้ำๆ และไม่เปลืองเครดิตเรียก API เปรียบเสมือนการจัดสรรทรัพยากรให้ถูกคน ถูกงาน ในทีมพัฒนาซอฟต์แวร์ครับ

---

## สไลด์ที่ 3 — Agent มีกี่ตัว

ระบบ Oh My OpenAgent จัดโครงสร้าง Agent ออกเป็นสามระดับชั้น คล้ายกับองค์กรที่มีพนักงานหลายระดับ ระดับแรกคือ Agents ระดับสูง ประกอบด้วย Sisyphus ที่เป็นหัวหน้างานเริ่มต้น Prometheus ที่ทำหน้าที่เหมือนสถาปนิกหรือนักวางแผน Atlas ที่เป็นผู้ปฏิบัติงานตามแผน และ Hephaestus ที่เป็นผู้เชี่ยวชาญระดับสูงที่คิดวิเคราะห์เชิงลึก ระดับที่สองคือ Core Agents ที่ทำหน้าที่แบ่งประเภทงานย่อย เช่น core research สำหรับค้นคว้าหาข้อมูล core explore สำหรับสำรวจโค้ดเบส และ core development สำหรับงานพัฒนาโค้ดโดยเฉพาะ ระดับที่สามคือ Tools ซึ่งเป็นเครื่องมือลงมือทำจริง เช่น terminal สำหรับรันคำสั่งเทอร์มินัล file สำหรับอ่านและเขียนไฟล์ และ code execution สำหรับรันโค้ดทดสอบความถูกต้อง สิ่งสำคัญคือ เราไม่จำเป็นต้องเรียกใช้ Core Agents หรือ Tools เอง เพราะ Agents ระดับสูงจะตัดสินใจใช้งานพวกเขาเองอัตโนมัติตามความเหมาะสมครับ

---

## สไลด์ที่ 4 — กฎทอง

กฎทองที่ต้องจำไว้มีอยู่ว่า ลำดับความสำคัญในการตัดสินใจคือ Agents มาก่อน Core แล้วค่อยถึง Tools เปรียบเสมือนในองค์กร ผู้จัดการโครงการคือคนตัดสินใจว่าจะส่งงานไปให้ทีมไหน หัวหน้าทีมแต่ละฝ่ายเป็นคนแบ่งงานย่อย และพนักงานปฏิบัติการเป็นคนลงมือทำจริง ถ้าคุณลูกค้าเดินเข้ามาสั่งงานตรงๆ กับพนักงานปฏิบัติการโดยไม่ผ่านหัวหน้างาน งานอาจออกมาผิดทิศทาง ไม่สอดคล้องกับเป้าหมายใหญ่ของโครงการ ดังนั้นหลักการสำคัญคือ อย่าเลือก Tools เองด้วยตรง และอย่าพยายามบังคับ Core Agents ให้ทำงานนอกลู่นอกทาง ให้ Agents ระดับสูงเป็นผู้ตัดสินใจสั่งการ ระบบจะจัดสรรทรัพยากรได้อย่างมีประสิทธิภาพที่สุด และคุณจะได้ผลลัพธ์ที่ดีที่สุดครับ

---

## สไลด์ที่ 5 — Sisyphus ตัวเริ่มต้น

มาทำความรู้จักกับ Sisyphus กัน Sisyphus เป็น Agent เริ่มต้นที่คุณจะพบโดยอัตโนมัติเมื่อเปิด OpenCode TUI เขาทำหน้าที่เหมือนพนักงานอัตราจ้างที่พร้อมทำงานง่ายๆ ได้ทันที คุณควรใช้ Sisyphus เมื่องานไม่ซับซ้อน เช่น การแก้ไขไฟล์เดียว การแก้คำผิด พิมพ์ผิด หรือการถามคำถามที่มีคำตอบตรงๆ วิธีการใช้งานง่ายมาก เพียงเปิด OpenCode แล้วพิมพ์คำสั่งเป็นภาษาธรรมชาติได้เลย ยกตัวอย่างเช่น ช่วยแก้ไขฟังก์ชันนี้ให้รองรับกรณีที่ค่าเป็น null ด้วย หรือ ช่วยเพิ่ม comment อธิบายโค้ดส่วนนี้หน่อย นอกจากนี้ยังมีทางลัดสำคัญที่ควรรู้จัก คือคำสั่ง ultrawork หรือ ulwเมื่อพิมพ์คำสั่งนี้ Sisyphus จะเข้าสู่โหมดทำงานอัตโนมัติ โดยที่เขาจะพยายามทำงานให้เสร็จเองโดยไม่ต้องถามคุณยืนยันทุกขั้นตอน เหมาะกับงานที่คุณมั่นใจว่าถ้าส่งให้เขาทำแล้วจะไม่มีอะไรผิดพลาดร้ายแรงครับ

---

## สไลด์ที่ 6 — Prometheus ผู้วางแผน

ถ้า Sisyphus คือช่างซ่อมที่มาซ่อมของเสีย Prometheus ก็คือสถาปนิกที่ออกแบบบ้านทั้งหลังก่อนก่อสร้าง คุณควรใช้ Prometheus เมื่อเผชิญหน้ากับงานใหญ่ที่ไม่แน่ใจขอบเขต งานที่ต้องวางแผนก่อนลงมือ หรืองานที่ต้องการให้มีคนช่วยแบ่งเป็นขั้นตอนที่ชัดเจน ขั้นตอนการใช้งาน Prometheus มีห้าขั้นตอนหลักๆ ขั้นตอนแรก กดแป้น Tab ที่ prompt แล้วเลือก Prometheus จากรายชื่อ Agent ขั้นตอนที่สอง ตอบคำถาม Interview ที่ Prometheus ถามเพื่อกำหนดขอบเขตงานให้ชัดเจน เช่น งานนี้ต้องการแก้ไขไฟล์ไหนบ้าง มีข้อจำกัดอะไร ต้องการผลลัพธ์แบบไหน ขั้นตอนที่สาม Prometheus จะวิเคราะห์ข้อมูลที่คุณให้ และสร้างแผนงานที่ครอบคลุมออกมาให้คุณดู ขั้นตอนที่สี่ คุณต้องอนุมัติแผนที่เขาสร้างมา หรือขอให้ปรับแก้หากมีส่วนไหนไม่ถูกใจ และขั้นตอนสุดท้าย พิมพ์คำสั่ง start work เพื่อให้ Prometheus ส่งต่อแผนงานให้ Atlas ลงมือทำทันที ข้อดีของการวางแผนก่อนคือ คุณจะได้แผนที่ครอบคลุม ไม่ลืม edge case สำคัญ และรู้ตั้งแต่ต้นว่าต้องทำอะไรบ้าง ก่อนที่จะมีการแก้ไขเกิดขึ้นจริงครับ

---

## สไลด์ที่ 7 — Hephaestus ผู้คิดลึก

ถ้างานที่คุณเจอต้องการความคิดเชิงลึก การวิเคราะห์ซับซ้อน หรือการแก้ไขปัญหาที่ Agent ทั่วไปทำไม่สำเร็จ นั่นเป็นสัญญาณว่าคุณต้องการ Hephaestus Hephaestus เป็น Agent ระดับเชี่ยวชาญที่ถูกออกแบบมาสำหรับการคิดวิเคราะห์เชิงลึกโดยเฉพาะ สัญญาณที่บ่งบอกว่าควรเรียกเขามาช่วยมีอยู่สามประการหลักๆ ประการแรก เมื่อ Agent ตัวอื่นวนเวียนอยู่กับปัญหาและไม่สามารถหาสาเหตุที่แท้จริงเจอ ประการที่สอง เมื่อคุณต้องค้นคว้าเทคโนโลยีหรือโดเมนที่คุณไม่คุ้นเคย เช่น การทำงานกับ library ใหม่ หรือ architecture ที่ซับซ้อน และประการที่สาม เมื่อคุณต้องออกแบบระบบใหม่หรือรีแฟกเตอร์โค้ดที่กระทบหลายส่วนในโครงการพร้อมกัน วิธีการเรียกใช้ Hephaestus ทำได้โดยการกดแป้น Tab ที่ prompt แล้วเลือก Hephaestus จากนั้นอธิบายปัญหาหรือคำถามที่คุณต้องการให้เขาวิเคราะห์อย่างละเอียด เขาจะใช้เวลาคิดวิเคราะห์อย่างถี่ถ้วนก่อนที่จะเสนอทางออกที่ดีที่สุดครับ

---

## สไลด์ที่ 8 — ตารางตัดสินใจ

ตอนนี้เรามาดูตารางตัดสินใจกันว่าในแต่ละสถานการณ์ควรใช้ Agent ตัวไหน ถ้าเป็นงานเล็กน้อย เช่น แก้คำผิด พิมพ์ผิด หรือแก้ไขเพียงไฟล์เดียว ให้ใช้ Sisyphus ตรงๆได้เลย ไม่ต้องเสียเวลาวางแผน ถ้าเป็นฟีเจอร์ใหม่หรืองานขนาดกลางที่ต้องแก้ไขหลายไฟล์ ให้เริ่มจาก Prometheus เพื่อวางแผน แล้วส่งต่อให้ Atlas ลงมือทำ ถ้าเป็นการรีแฟกเตอร์ระบบใหญ่หรือเปลี่ยนแปลงโครงสร้างครั้งสำคัญ ให้เริ่มจาก Prometheus ส่งให้ Atlas และปรึกษา Hephaestus ในการออกแบบส่วนที่ซับซ้อน ถ้าคุณยังไม่รู้ว่าต้องทำอะไรดี ให้ Prometheus เป็นคนสัมภาษณ์คุณ เพื่อช่วยกำหนดขอบเขตงานให้ชัดเจน ถ้าติดปัญหาดีบักลึกหรือต้องการค้นคว้าข้อมูลเชิงลึก ให้ใช้ Hephaestus โดยเฉพาะ และถ้าคุณขี้เกียจอธิบายรายละเอียดมาก หรืองานไม่ซับซ้อนแต่อยากให้ทำเสร็จเอง ให้ใช้คำสั่ง ultrawork ใน Sisyphus เพราะเขาจะพยายามทำงานให้เสร็จโดยอัตโนมัติครับ

---

## สไลด์ที่ 9 — สิ่งไม่ควรทำ

มาพูดถึงข้อห้ามสำคัญที่มือใหม่มักจะทำผิดกัน ข้อแรก อย่าเริ่มต้นที่ Atlas โดยตรงถ้าคุณยังไม่มีแผนงานที่ชัดเจน เพราะ Atlas ทำหน้าที่เป็นผู้ปฏิบัติงานตามแผน ไม่ใช่ผู้วางแผน ถ้าเขาไม่มีแผน งานที่ออกมาจะกระจัดกระจายและไม่เป็นไปตามเป้าหมาย ข้อที่สอง ไม่ต้องเรียก Oracle เองด้วยตรง เพราะ Oracle จะถูกเรียกโดย Sisyphus หรือ Atlas โดยอัตโนมัติเมื่อพวกเขาต้องการค้นหาความรู้ในระบบ การเรียกเองอาจทำให้เกิดความซ้ำซ้อน ข้อที่สาม อย่าพยายามเลือก Core Agents หรือ Tools เอง เช่น ไม่ต้องบอกว่าให้ใช้ core research หรือ terminal ให้ Agents ระดับสูงเป็นคนตัดสินใจว่าจะใช้ใคร และข้อสุดท้าย อย่าใช้ ultrawork กับงานที่ต้องการความแม่นยำสูงหรือมีผลกระทบร้ายแรงหากผิดพลาด เพราะโหมดนี้จะทำงานอัตโนมัติโดยไม่ถามยืนยันคุณก่อนครับ

---

## สไลด์ที่ 10 — สรุปง่ายๆ

มาสรุปสิ่งสำคัญที่ต้องจำกันอีกครั้ง ข้อหนึ่ง ถ้าเป็นงานเล็กน้อย แก้ไขง่ายๆ ให้ใช้ Sisyphus พิมพ์คำสั่งตรงๆ ได้เลย ข้อสอง ถ้าเป็นงานขนาดกลางที่ไม่ซับซ้อนมากนัก ให้ใช้ Sisyphus ควบคู่กับคำสั่ง ultrawork เพื่อให้เขาทำงานอัตโนมัติจนเสร็จ ข้อสาม ถ้าเป็นงานใหญ่ที่ต้องวางแผนละเอียด ให้ใช้ Prometheus เพื่อวางแผน แล้วส่งต่อให้ Atlas ลงมือทำ และถ้างานนั้นต้องการการวิเคราะห์เชิงลึกเพิ่มเติม ให้ Hephaestus ช่วยคิดเสริม หลักการง่ายๆ ที่อยากให้จำไว้คือ เลือก Agent ให้ถูกตัว งานก็จะเสร็จไว ไม่เสียเวลา และได้ผลลัพธ์ที่ดีที่สุด ขอบคุณที่รับชมและรับฟังคู่มือนี้จนจบ หวังว่าจะช่วยให้คุณใช้ OpenCode Agents ได้อย่างมีประสิทธิภาพมากขึ้นครับ
